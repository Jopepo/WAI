# WAI 3 Privacy Data Map

Status: engineering source of truth for the unreleased WAI 3 branch.

This document separates data collected by the WAI service from data that stays
inside the app. It must be reviewed against the deployed Supabase project and
App Store Connect answers before WAI 3 is released.

## Service-side data

| Data | Source | Purpose | Destination | Retention/deletion | App Store category |
| --- | --- | --- | --- | --- | --- |
| Apple provider subject | Sign in with Apple | Authenticate the account and bind deletion reauthentication | Apple and Supabase Auth | Until account deletion, subject to provider records | User ID |
| Email address, when Apple supplies it | Sign in with Apple | Account authentication and contact identity | Apple and Supabase Auth | Until account deletion, subject to provider records | Email Address |
| Supabase user UUID | Supabase Auth | Identify the account, profile, cache owner, and authorization boundary | Supabase Auth and `wai_profiles` | Hard-deleted by the in-app account deletion flow | User ID |
| Approval code and access status timestamps | Generated by WAI backend | Manual approval, revocation, and offline authorization age | `wai_profiles` | Profile row cascades when the account is deleted | User ID / App Functionality |
| Crew name, employee/TAP number, and company email | Optional approval email sent by the user from Mail | Verify eligibility before approval | Developer support mailbox, not the WAI API | Release blocker: define and publish mailbox retention/deletion procedure | Name, Email Address, User ID |
| Network request metadata | Authentication and protected data requests | Deliver and protect the service | Apple, Supabase, and their infrastructure providers | Release blocker: verify deployed provider logging and retention settings | Review before App Store submission |
| Origin and destination ICAO codes | User opens a flight briefing | Retrieve current METAR data | US National Weather Service Aviation Weather Center | Not retained by WAI; provider request logs are subject to provider policy | Not personal data |

The app sends authentication credentials only to Apple and the configured
Supabase project over HTTPS. It sends an Apple authorization code to the
server-only account deletion function only after a fresh deletion
reauthorization. Service-role credentials and the Apple private key never ship
in the app.

## Device-only data

| Data | Source | Local storage | Leaves device through WAI | Deletion |
| --- | --- | --- | --- | --- |
| Calendar titles, event titles, notes, dates, and identifiers used to recognize the roster | EventKit, after explicit Calendar permission | Parsed into the encrypted, owner-bound roster cache | No | Sign-out, account deletion, owner mismatch, or roster replacement |
| Imported iCalendar roster | User-selected file | Parsed into the encrypted, owner-bound roster cache | No | Sign-out, account deletion, owner mismatch, or roster replacement |
| Roster duties, legs, rest analysis, automatic wake-up and pickup times | Local calculation | Encrypted, owner-bound roster cache | No | Sign-out or account deletion |
| Room numbers | User input | This-device-only Keychain | No | User removal, sign-out, or account deletion |
| Manual calculation history and hotel stays | User input and local calculation | AES-GCM encrypted, owner-bound files with complete file protection | No | User removal, sign-out, or account deletion |
| Downloaded operational rules, hotel map, and What's New data | Protected WAI service | AES-GCM encrypted, owner-bound cache | No automatic onward transfer; a hotel search/contact value leaves WAI only when the user explicitly chooses an external Maps, phone, email, or browser action | Sign-out, account deletion, invalid owner, or cache replacement |
| Access and refresh tokens, offline approval grant, encryption keys, account-deletion intent, and local sign-out intent | Apple/Supabase and local security state | Non-synchronizing, this-device-only Keychain | Tokens are sent only to the configured authentication/service endpoints | Sign-out, account deletion, revocation handling, or invalid state |
| Current METAR observations | Aviation Weather Center | Memory only while the flight screen is open | No onward transfer | Discarded with the view/app process |

Protected files use complete file protection and are excluded from device
backup. The WAI 3 runtime does not load roster, room, calculation, or hotel data
from the legacy unprotected stores. On secure startup it removes the legacy WAI
2.2 personal-data keys, the three plaintext operational JSON cache files, and
old URL-cache responses. Private WAI network requests use an ephemeral session
without cookies, URL credentials, or a response cache. Authentication and
protected-data responses are streamed into bounded buffers; an advertised or
actual response over the applicable hard limit is rejected before decoding or
storage.

Opening a flight briefing requests current METAR observations from the US
National Weather Service Aviation Weather Center. The request contains only the
four-letter ICAO codes for the displayed origin and destination. It does not
contain roster contents, account identifiers, location-services data, room
numbers, briefing inputs, or commander passwords. Failure or lack of network
access leaves the rest of the briefing available offline.

EventKit is queried only for a bounded roster search window. WAI does not copy
every returned calendar event into its parser or encrypted archive: it first
requires bounded titles and identifiers, valid dates, and notes matching a
known roster-event shape. Candidate calendars must then identify themselves as
TAP calendars or contain a recognized flight fingerprint. The resulting roster
snapshot remains on the device.

When roster imports overlap, the newest import is authoritative only inside its
declared coverage. It replaces duties and issues in that period, including
removing duties cancelled by the newer roster, while older duties outside the
new coverage remain available. Original encrypted import segments are retained
locally so this resolution is deterministic.

Sign-out and account deletion write a device-only interruption marker before
destructive work begins. Local authorization and protected personal data are
cleared before WAI waits for remote sign-out. If the app terminates during that
operation, the next launch completes local cleanup before attempting to restore
access. Session-generation and cache-access fences also invalidate in-flight
authentication, refresh, and write operations, preventing a late response from
recreating protected state after sign-out or account deletion has begun.

When the secure app is inactive or in the background, WAI covers its interface
with a neutral privacy view so roster, hotel, room, and timing details are not
left in the normal app-switcher snapshot. This does not prevent a user from
taking a screenshot while actively viewing the app.

## Not collected by WAI 3

- Advertising identifiers.
- Analytics or advertising events.
- Precise or coarse location from Location Services.
- Contacts, photos, microphone, camera, or health data.
- Calendar or roster contents on the WAI backend.
- Room numbers, manual calculations, hotel stays, wake-up times, or pickup times
  on the WAI backend.

## Privacy manifest mapping

`WAI/PrivacyInfo.xcprivacy` currently declares:

- Name, Email Address, and User ID as linked to the user, used for App
  Functionality, and not used for tracking.
- UserDefaults reason `CA92.1` for app-owned preferences and legacy-data cleanup.
- File Timestamp reason `C617.1` for app-owned protected file metadata.
- Tracking disabled and no tracking domains.

Calendar permission is separately described by
`NSCalendarsFullAccessUsageDescription`. Roster content is processed locally
and is not sent to the WAI backend. When the user saves a custom briefing
flight time, WAI creates or updates a leg event in the selected Calendar
account; that account's provider may sync the event off-device. The event
contains the flight number, route, departure, duration, and an opaque WAI
marker. Passenger data and the commander password are never added to Calendar.
This user-initiated write path must be reassessed against App Store privacy
definitions before release.

## Release blockers

1. Deploy a dedicated production Supabase project and audit auth, database,
   storage, Edge Function, and infrastructure logs.
2. Define approval-mailbox retention and deletion handling.
3. Confirm the final public privacy-policy URL and make it reachable without
   login.
4. Replace the public WAI 2.2 policy only when WAI 3 is ready to ship.
5. Match App Store Connect privacy answers to this map and the deployed service.
6. Generate and review Xcode's privacy report from the final archive.
7. Prove that the final WAI 3 app bundle contains no operational JSON, public
   repository URL, secret, service-role key, or Apple private key.
